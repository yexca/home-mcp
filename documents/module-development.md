# Module Development

Modules add capabilities without changing the transport layer or the core
dispatcher. Each module owns its MCP schemas, service logic, and provider
adapters.

## Required Layout

```text
modules/<name>/
  __init__.py
  manifest.py
  schemas.py
  service.py
  providers/
    __init__.py
tests/
  test_<name>.py
```

`manifest.py` is the startup entry point. It must expose:

```python
def register_tools(registry, settings) -> None:
    ...
```

The function should return without registering tools when the module is not
enabled.

## Tool Definition Rules

Use `tools.registry.ToolDefinition`:

```python
registry.register(
    ToolDefinition(
        name="example_action",
        title="Example Action",
        description="Do one clearly scoped action.",
        input_schema=EXAMPLE_INPUT_SCHEMA,
        output_schema=EXAMPLE_OUTPUT_SCHEMA,
        risk_level="medium",
        handler=example_action,
        creates_job=True,
    )
)
```

Rules enforced by the registry:

- Tool names must be lowercase snake_case.
- `risk_level` must be `low`, `medium`, or `high`.
- The input schema root must be a JSON object.
- Schema fields must not include sensitive names such as `api_key`, `token`,
  `base_url`, `authorization`, `access_token`, or `path`.
- Handlers must be callable and may be sync or async.

## Service Rules

Service handlers receive validated arguments and a `RequestContext`.

Use the context for:

- `ctx.config`: merged settings.
- `ctx.caller`: resolved caller identity.
- `ctx.artifacts`: artifact creation and reads.
- `ctx.jobs`: job progress updates when needed.
- `ctx.policy`: additional domain-specific checks if needed.
- `ctx.audit`: audit helpers.
- `ctx.limits`: per-window rate limits.
- `ctx.job_id`: current job ID when `creates_job=True`.

Do not bypass `ToolDispatcher` for normal tool execution. The dispatcher owns
validation, caller resolution, policy, jobs, audit, and failure shaping.

## Provider Rules

Provider adapters should live under `modules/<name>/providers/` and normalize
external APIs into internal dataclasses or simple dictionaries.

Provider adapters should:

- Read secrets only from settings that were populated by environment variables.
- Convert provider errors into `GatewayError` stable codes.
- Hide provider-specific URLs and tokens from MCP responses.
- Return bytes or normalized metadata to service code.
- Leave artifact persistence to the service layer.

## Artifacts

Any generated file should be persisted through `ctx.artifacts`.

Use `create_from_bytes()` for generated bytes and `create_from_file()` for local
files already produced by a trusted internal step. Artifact metadata should be
small and non-secret. Include provider names, selected options, model names, and
usage summaries only when safe.

Artifact kinds currently include:

- `image`
- `audio`
- `document`
- `print`
- `temp`

## Jobs

Set `creates_job=True` when the tool:

- Calls an external provider.
- Creates or transforms an artifact.
- Sends a Matrix message.
- Submits a print job.
- Performs any user-visible side effect.

The dispatcher creates and finishes jobs automatically. Service code can call
`ctx.jobs.update_progress(ctx.job_id, progress, summary)` for longer workflows.

## Policy And Risk

Use `high` risk for tools that send data outside the gateway, trigger physical
actions, or affect external accounts. Current examples are Matrix sends and
print submission.

Use module-level allowlists in service code for domain constraints such as room
IDs and printer IDs. The central policy engine already checks current Matrix
and printer constraints, but service-level validation keeps direct unit tests
and future refactors safe.

## Configuration Validation

Add module validation in `app.config._validate()` when enabling a module
requires specific settings. Validation should fail at startup for missing
provider URLs, missing API keys, invalid defaults, empty allowlists, or unknown
provider names.

Keep configuration examples secret-free. Use `${ENV_NAME}` placeholders for
values that must come from the environment.

## Tests To Add

For every new module or provider behavior, add focused tests for:

- Registration only when enabled.
- Tool schema contract and forbidden schema fields.
- Successful dispatch through `ToolDispatcher`.
- Policy denial for anonymous or unauthorized callers.
- Provider error mapping.
- Artifact ownership and metadata when files are created.
- Rate limits or allowlists when applicable.
